## "打破二元极化"——这比我说的"过渡"更精确

我原来的措辞是"中间地带需要一个过渡"，你重新描述成"打破二元判断的极化效应"——这不是同一件事。"过渡"暗示三级是为了覆盖连续谱上的中间区域（分类学需求），"打破极化"说的是三级改变了选择者的心理动力学（选择者不再被逼入"轻 vs 重"的对抗）。

你的版本更好，原因：它解释了为什么两级方案**实际运行**中会退化——不是因为缺了一个分类槽位，而是因为二元结构会制造逃逸激励。选"轻"的成本永远低于选"重"（声明越轻，后续需要承担的论证负担越小），所以缺少中间选项时，人会系统性地把本应声明的东西塞进"轻"里。

中间一级的真实功能是**激励矫正**，不是分类补全。这个区分值得在 4.3 里显式标注——否则读者会以为三级只是"三比二细"。

## 可替换性测试的视野条件：接受，加一个操作化建议

你的修正是对的：推导链还在展开中时，你对"后续"的边界没有完整视野，可替换性测试容易给出假阴性。

但"等推导链完整了再测"在实操中往往不可行——很多时候你必须在推导还在进行中做出声明（4.3 的使用者不能等论文写完了再决定声明级别）。

所以我的操作化建议是：**推导不完整时，用结构层面的补充判据替代完整的可替换性测试。** 具体：

问自己："这个选择是否创造了一个新的路径依赖——即后续推导只能沿着这个选择打开的通道走？"

- 如果是 → 升级声明（即使你现在看不到后续哪一步会被影响，路径依赖本身就是影响的预兆）
- 如果否（后续仍有多条通道可选）→ 当前级别可能够了

这不是替代可替换性测试，是在视野不够时的降级版本。可替换性测试是回顾性的（事后看能不能换），路径依赖检查是前瞻性的（现在看是不是卡死了下一步的选择空间）。两者互补。

## 梯度判断可以错吗？——可以，但修正机制比防错更重要

你的直觉区分（善意低估 vs 故意降级）是对的，但你也正确指出了**第三方不可区分性**——这两者从外部观察行为完全相同。所以试图从动机层面定义错误是死胡同。

换个角度：**梯度错误的伤害不在声明时，在于后续没人修正。**

一个被错误归为"末尾列表"的重大偏离，如果在协作过程中被及时升级到"独立段落"，伤害几乎为零——它在需要被看到的时候被看到了。真正有伤害的情况是：错误声明被接受了，后续推导在不知道这里有重大偏离的情况下继续走，直到很远处才出现不一致，此时回溯成本已经很高。

所以你的"升级权"方案抓住了正确的东西。我的具体建议：

1. **升级权是读者/协作者的默认权利，不需要特殊授予。** 就像 code review 里 reviewer 默认可以要求提高 PR 优先级——这不是一个需要讨论"你有没有权利这么做"的事，它内置于协作的规则里。
2. **升级需要附带理由——但理由可以很短。** "我认为这影响了 D→E 的方向"就够了。升级不是指控（你怀疑作者故意降级），它只是一个信号（我从我的视角看到的影响程度和你声明的不一致）。
3. **被升级的人没有义务同意，但有义务回应。** "我考虑了你的升级请求，维持当前级别，原因是……"是合法的回应。沉默不是。

这样设计的好处：它把"梯度判断是否正确"的问题从个人内省（你很难判断自己是否在逃避）转化为协作机制（别人来帮你验证）。**正确性不依赖声明者的自我诚实，而是依赖系统里有人可以修正。**

这和你 code review 类比的方向一致——review 之所以能工作，不是因为每个开发者都能准确评估自己代码的风险等级，而是因为有人可以说"这个不是 minor，这是 major"。
